# 헌터x헌터 프로젝트 실전 Git 활용 가이드

이 문서는 Git이 익숙하지 않은 백엔드/프론트엔드 개발자를 위해 **가장 자주 사용하는 필수 Git 명령어**와 **상황별 브랜치 관리 및 통합 시나리오**를 알기 쉽게 정리한 개발 문서입니다.

---

## 1. Git의 핵심 3단계 흐름 이해하기

코드 변경사항은 내 컴퓨터에서 원격 저장소(GitHub 등)까지 총 4개의 단계를 거쳐 올라갑니다.

```text
 [ 내 컴퓨터 (작업 공간) ] ➔ ➔ ➔ ➔ ➔ [ 원격 저장소 (GitHub) ]
 ┌──────────────────┬─────────────────┬─────────────────┐               ┌──────────────────┐
 │  Working Directory│  Staging Area   │  Local Repo (.git)│  (Push)     │   Remote Repo    │
 │ (현재 수정 중인 코드)│ (제출 대기 상태)  │ (내 컴퓨터에 저장) │ ───────────►│ (인터넷 저장소)  │
 └──────────────────┴─────────────────┴─────────────────┘               └──────────────────┘
         │ (git add)        │ (git commit)             
         ▼                  ▼                          
```

---

## 2. 반드시 알아야 할 필수 Git 명령어 백과사전

### 🔍 1) 상태 및 히스토리 확인
* **현재 내 깃 상태 보기**: `git status`
  * 내가 어떤 파일을 수정했는지, 어떤 브랜치에 있는지, 추가되지 않은 새 파일이 무엇인지 실시간으로 보여줍니다. (가장 자주 치는 명령어)
* **커밋 로그(기록) 보기**: `git log --oneline`
  * 지금까지 저장(Commit)된 히스토리를 한 줄씩 깔끔하게 보여줍니다.

### 💾 2) 코드 변경사항 저장하기
* **수정한 파일 제출 대기선에 올리기**: `git add .`
  * 현재 폴더 내의 모든 수정된 파일을 Staging Area(제출 대기선)에 올립니다. 
  * 특정 파일만 올리려면 `git add backend/src/main/.../Service.java` 형태로 파일명을 씁니다.
* **로컬 저장소에 최종 저장하기**: `git commit -m "타입: 메시지"`
  * 대기선에 올라간 파일들을 패키징하여 내 컴퓨터 저장소에 최종 영구 기록합니다.

### 🌿 3) 브랜치 제어하기
* **현재 브랜치 목록 확인**: `git branch`
* **새로운 브랜치 만들고 즉시 이동**: `git checkout -b {브랜치명}`
* **기존에 존재하던 다른 브랜치로 이동**: `git checkout {브랜치명}`
* **쓸모없어진 로컬 브랜치 삭제**: `git branch -d {브랜치명}`

### ☁️ 4) 원격 저장소(GitHub)와 주고받기
* **원격 저장소의 최신 코드 내 컴퓨터로 가져오기**: `git pull origin {브랜치명}`
* **내 컴퓨터의 커밋 내용을 원격 저장소로 업로드**: `git push origin {브랜치명}`

## 3. 상황별 실전 가이드 & 워크플로우

### 🎬 상황 A: "오늘 개발 시작하려고 컴퓨터를 켰을 때"
어제 다른 팀원이 올렸거나, 내가 다른 곳에서 수정한 최신 코드가 원격 저장소에 있을 수 있으므로 항상 작업을 시작하기 전에 내 로컬 `develop` 브랜치를 최신으로 갱신해야 코드가 꼬이지 않습니다.

```bash
# 1. 내 로컬의 develop 브랜치로 이동합니다.
git checkout develop

# 2. 원격 저장소(GitHub)의 최신 코드를 내 컴퓨터로 당겨옵니다.
git pull origin develop
```

---

### 🎬 상황 B: "새로운 기능(예: 번역기 사전 필터)을 개발해야 할 때"
절대로 `develop` 브랜치에 직접 코드를 작성하지 마세요! 작업 시작 전 전용 피처 브랜치를 생성해 들어갑니다.

```bash
# 1. develop이 최신인 상태에서 새로운 브랜치를 생성하고 이동합니다.
git checkout -b feature/translator-dictionary

# 2. 텍스트 에디터(VS Code 등)에서 코딩을 열심히 진행합니다.

# 3. 수정한 파일들을 확인하고 저장합니다.
git status
git add .
git commit -m "feat: 헌터어 번역 고유명사 사전 필터링 구현"
```

---

### 🎬 상황 C: "방금 만든 기능을 개발 완료해서 통합(develop)하고 싶을 때"
로컬 피처 브랜치에서 코딩과 자체 테스트를 마치고 이를 동료들과 공유하기 위해 통합하는 프로세스입니다.

```bash
# 1. 내가 만든 피처 브랜치를 원격(GitHub)에 업로드(Push)합니다.
git push origin feature/translator-dictionary

# 2. GitHub 저장소 홈페이지로 접속합니다.
# 3. 상단에 뜬 [Compare & pull request] 초록색 버튼을 클릭합니다.
# 4. 대상 브랜치가 "base: develop" ◀ "compare: feature/translator-dictionary" 인지 확인합니다.
# 5. PR 메시지를 작성하고 [Create pull request]를 누릅니다.
# 6. 코드 검증이 완료되면 GitHub 웹페이지에서 [Merge pull request]를 눌러 합칩니다.

# 7. 합치기 완료 후, 내 컴퓨터로 돌아와 로컬 브랜치를 정리합니다.
git checkout develop
git pull origin develop                       # 합쳐진 최신 버전을 다시 가져옴
git branch -d feature/translator-dictionary   # 볼일 다 본 로컬 브랜치 삭제
```

---

### 🎬 상황 D: "작업하던 중 급한 버그가 생겼는데, 작성 중인 코드가 지저분해서 Commit하기 아까울 때"
현재 브랜치에서 열심히 짜던 임시 코드를 잠시 서랍에 넣어 보관하고, 깨끗한 상태로 다른 브랜치로 가서 일을 처리한 뒤 다시 돌아와 꺼내 쓰는 유용한 기능입니다.

```bash
# 1. 수정 중이던 코드를 임시 임시 저장소(서랍)에 넣어 숨겨둡니다.
git stash

# 2. 작업 공간이 깨끗해졌으므로 안심하고 급한 버그가 있는 브랜치로 이동해 수정합니다.
git checkout develop
# (급한 일 처리 완료...)

# 3. 다시 내가 작업하던 브랜치로 돌아옵니다.
git checkout feature/translator-dictionary

# 4. 서랍에 보관해 두었던 임시 코드를 다시 꺼내어 복구합니다.
git stash pop
```

---

### 🎬 상황 E: "앗! 방금 커밋을 했는데 메시지에 오타가 났거나 파일을 빼먹었을 때"
가장 직전에 내린 커밋을 취소하지 않고, 살짝 내용을 얹어 수정하는 방법입니다.

```bash
# 만약 깜빡하고 안 올린 파일이 있다면
git add 깜빡한파일.java

# 직전 커밋을 수정 모드로 덮어씁니다. (커밋 메시지도 수정 창이 열립니다.)
git commit --amend
```

---

### 🎬 상황 F: "코드를 고치다가 완전히 꼬여서 마지막 커밋(완벽했던 상태)으로 되돌리고 싶을 때"
수정했던 모든 내용을 지우고 가장 최신의 깨끗했던 커밋 시점으로 내 작업 공간을 초기화합니다. **(주의: 수정한 내용이 완전히 날아갑니다!)**

```bash
# 변경 사항을 모두 취소하고 마지막 커밋 상태로 되돌리기
git reset --hard HEAD
```
